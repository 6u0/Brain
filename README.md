# 脳髄XR体験（仮称：Dogra Magra XR）システム仕様書

## 1. プロジェクト概要
本作は「現実からの乖離」「身体と自己の切断」をテーマにした物理ギミック連動型のXR（VR/MR）体験作品である。
体験者は1.8m四方の個室でHMDと自作の「脳みそデバイス」「ヘッドセットデバイス」を装着し、正木教授の語りを通じて自らの脳を投棄する体験を行う。

## 2. システムアーキテクチャ（データフロー）
すべてのデバイス間通信・演出制御は **PC（Unity）** を中央ハブとして統合管理される。デバイス同士の直接通信は行わない。

* **入力の流れ:** センサー群 -> ESP32 -> (Serial/OSC) -> PC (Unity)
* **処理・判定:** PC (Unity) 内のステートマシン（ExperienceManager）
* **出力の流れ:** PC (Unity) -> (Serial/OSC) -> ESP32 -> アクチュエータ群 / PC -> HMD (映像・音響)

## 3. ハードウェア仕様

### 3.1. 脳みそデバイス（Brain Device）
* **役割:** 触覚・位置入力、および鼓動の出力。
* **内蔵:** Meta Quest コントローラー（位置・回転トラッキング用）
* **入力 (ESP -> Unity):**
    * 圧力センサー ×6 (`float pressure[6]`)
    * 心拍センサー（オプション: `float bpm` / `float pulse_wave`）
* **出力 (Unity -> ESP):**
    * 鼓動アクチュエータ（ソレノイド or 振動スピーカー用PWM信号）

### 3.2. ヘッドセットデバイス（Headset Device）
* **役割:** 頭部への物理的フィードバック、視界ギミックの物理制御。
* **入力 (ESP -> Unity):**
    * 手侵入検知（IR/スイッチ等: `bool isHandInside`）
    * 測距センサー（ドーム内距離: `float domeDistance`）
    * ホールセンサー（血管ちぎれ検知: `bool isVesselTorn`）
* **出力 (Unity -> ESP):**
    * 頭皮押し込み用サーボ ×6 (`int servoAngle[6]`)
    * ミラー跳ね上げ用サーボ (`int mirrorServoAngle`)
    * 頭部バイブレーター (`bool isVibrating`)

## 4. ソフトウェア設計（Unity）

### 4.1. コアスクリプト構成 (アーキテクチャ)
単一責任の原則（SRP）に基づき、以下のManagerクラスで構成する。

1.  `ExperienceManager`: 全体の状態遷移（ステートマシン）を管理する総合司令塔。
2.  `ESPNetworkManager`: ESP32とのシリアル通信/OSC通信を担当。データのパースと送信。
3.  `HardwareController`: ESPからのデータを受け取り、Unity内のロジック（圧力の平均化など）に変換、およびデバイスへの命令を発行。
4.  `VFXManager`: パススルーのON/OFF、ホワイトアウト/ブラックアウト、SR映像（代替現実）の再生制御、ノイズエフェクトの描画。
5.  `AudioManager`: バイノーラル音声再生、ノイズ制御、環境音の停止（完全静寂の生成）。
6.  `PlayerController`: プレイヤーのVRカメラ（OVRCameraRig）の制御、強制移動（夢中遊行フェーズ）の制御。

### 4.2. 状態遷移表（ステートマシン）
`ExperienceManager` は以下の `Phase` Enumで状態を管理する。

| フェーズ | 状態名 | 開始トリガー | 主要な処理・演出 | 次フェーズへの遷移条件 |
| :--- | :--- | :--- | :--- | :--- |
| **Phase0** | `Wait` | アプリ起動時 | 待機状態。パススルーON。 | アテンド完了（手動トリガー） |
| **Phase1** | `Intro` | Phase0から遷移 | 案内役退場。正木教授の3Dモデル出現。頭部バイブ作動（頭蓋切開の錯覚）。語り開始。 | 一定時間経過 (20秒) |
| **Phase2** | `Contact` | Phase1終了 | 脳デバイスへの接触待機。圧力センサー値に応じたノイズ・頭部サーボのフィードバックループ実行。 | 脳を投げる（Questコンの速度閾値超過） or タイムアウト |
| **Phase3** | `Throw` | 投擲検知 | ミラー跳ね上げ(ESP)、画面ホワイトアウト→ブラックアウト、全音響停止。 | 一定時間経過 (物理ショック後の暗闇) |
| **Phase4** | `Erosion` | Phase3終了 | SR映像（代替現実）再生開始。プレイヤーの入力を無視し、オートで不規則なカメラ移動（夢中遊行）。 | 特定の座標・映像再生完了 |
| **Phase5** | `Ending` | Phase4終了 | カプセルへの溶解演出（VFX）。HMD着脱待機。 | HMD着脱検知（OVRPluginの近接センサー） |
| **Phase6** | `Complete`| HMD着脱 | 外部スピーカーから正木の声再生。アプリ終了処理。 | - |

## 5. Cursor / AI向け 実装ガイドライン

AIエディタが本プロジェクトのコードを生成・修正する際は、以下のルールを厳守すること。

1.  **コンポーネント指向:** `ExperienceManager` に全ての処理を記述せず、必ず `AudioManager` や `HardwareController` などの各Managerのメソッドを呼び出す設計にすること。
2.  **非同期処理:** 時間経過を伴う演出（フェーズ内の待機やフェード処理）は、基本的に `IEnumerator` (コルーチン) または `UniTask`（導入されている場合）を使用すること。
3.  **安全なフェールセーフ:** センサーの断線や体験者の想定外の行動（例：いつまでも脳を投げない）に備え、各フェーズには必ず `Timeout` 処理を実装し、進行不能バグを防ぐこと。
4.  **ハードコードの回避:** センサーの閾値（投擲速度 `brainThrowVelocityThreshold` など）やフェーズの待機時間は、Inspectorから調整できるように `[SerializeField]` 属性を付与すること。
5.  **Meta XR SDKの活用:** Questのコントローラー取得は `OVRInput`、パススルー制御は `OVRPassthroughLayer` の標準APIを使用すること。エディタ上でのパススルーバグ回避策も考慮すること。
